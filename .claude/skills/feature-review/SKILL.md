# feature-review

评审功能需求。**输出面向产品经理**，业务化表达，避免技术术语。

> 遵循 `github-comment` 规范。

## 成本估算标准

| 级别 | 工时 | 典型特征 |
|------|------|----------|
| **small** | < 1天 | 单文件改动、配置修改、简单 UI 调整 |
| **medium** | 1-3天 | 新增独立功能、涉及 2-5 个文件、需要单元测试 |
| **large** | 3-5天 | 跨模块改动、需要接口设计、涉及数据迁移 |
| **extra-large** | > 1周 | 架构变更、新增子系统、需要多方协调 |

## 影响范围分析

评估时考虑：
- **用户可见影响**: 哪些页面/功能会变化
- **数据影响**: 是否涉及数据结构变更、迁移
- **依赖影响**: 是否影响其他模块或下游服务
- **回滚难度**: 出问题能否快速回滚

## 模板

```markdown

| **业务价值** | {这个功能解决什么用户问题} |
| **预估成本** | 🟢 small / 🟡 medium / 🟠 large / 🔴 extra-large |

**主要挑战**:
- {用业务语言描述难点，如"需要对接第三方系统"}

**影响范围**:
- {哪些用户可见的功能会受影响}
- {是否涉及数据变更}

<details>

<details>
```
